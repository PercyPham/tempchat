# **Project: Ephemeral Encrypted Chat (Codename: "TempChat")**

## **1. Product Concept**

A zero-knowledge, short-lived chat web application designed for immediate, real-world social interactions. It facilitates group communication without requiring phone numbers, app store downloads, or permanent data footprints.

### **Key Use Cases**

- Short conferences or workshops.
- Temporary social gatherings (weddings, meetups).
- Secure, "in the moment" coordination between strangers.

## **2. Core Features**

- **Zero-Install:** Runs entirely in the mobile/desktop browser as a PWA.
- **QR-Based Entry:** Instant room joining via scanning.
- **Group Identity:** Every room has an immutable **Group Name** set at creation.
- **Zero-Knowledge Privacy:** The server facilitates and stores messages but cannot read them.
- **Automatic Self-Destruction:** Free rooms expire after 1 hour.
- **Persistence:** Encrypted messages are buffered in Redis; **late-joiners only see history generated after their specific join time.**
- **Monetization:** Boosts are additive — each purchased boost extends the room's remaining time by the boost's duration and raises participant/event caps to the boost's maximum (taking the higher of current and new values). Any room member can purchase a boost. Non-members attempting to join a full room can also boost it to gain entry, if the available boosts would raise the participant cap above the current count. Boost options are configured server-side and can change over time. Rooms start as Free at creation. See §6 for default creation values.

## **3. Security Architecture (The "Zero-Knowledge" Flow)**

### **3.1 Data Generation**

1. **Room ID (`roomId`):** Generated by the **server** upon creation request.
2. **Secret Key (`secret`):** Generated by the **client** browser. A high-entropy symmetric key (AES-256) used for message encryption.
3. **Room Access Key (roomAccessKey):** Generated by the **client** via `PBKDF2-HMAC-SHA-512(secret)` with salt `"rak"` and `600k+` iterations. This is the "root of trust" shared with the server once at room creation but never sent again in plaintext.
4. **Group Name:** A plaintext string provided by the creator and stored on the server.

### **3.2 User Identity**

1. **User ID (`userId`):** Generated by the **server** when a participant successfully joins a room.
2. **Display Name:** Chosen by the user and linked to their server-generated `userId`.
3. **Registry:** Per-user metadata (name, join timestamp, `join_eid`) is stored in `room:{roomId}:user:{userId}:meta`. All joined user IDs are tracked in `room:{roomId}:users`.
   - When a user joins, the server records the current `event_seq` value as that user's `join_eid` — the lower bound for history fetches.
   - Once the `max_participants` cap is reached, no more joins are accepted. When a join is rejected due to capacity, the client fetches the room's current `memberCount` and `maxParticipants` via `GET /api/rooms/:roomId`, then fetches available boost options via `GET /api/boost-options`. If any boost option offers a higher `maxParticipants` than the room's current cap, the client shows boost purchase options. Non-member boost requests use `uid: null` auth — the same as the initial join request.

### **3.3 Message Encryption & Request Signing**

To prevent replay attacks and unauthorized access, TempChat uses a signed-claim mechanism:

1. **Message Encryption:** Client encrypts `Plaintext` + `secret` → `Ciphertext` (AES-GCM).
2. **Auth Token Generation:**
   - **Claims:** The client constructs a JSON object: `{ "rid": roomId, "uid": userId, "ts": current_timestamp }`.
   - **Signing:** The client generates a signature using `HMAC-SHA-256(Claims, roomAccessKey)`.
   - **Envelope:** The `authToken` sent in the request header is a Base64-encoded string containing both the `Claims` and the `Signature`.

3. **Server Validation:**
   - **Signature Verification:** Server retrieves the stored `roomAccessKey` for the `roomId` and re-calculates the HMAC of the received claims.
   - **Drift Check:** The server verifies that the `ts` (timestamp) in the claims is within a **±5 second window** of the current server time to prevent replayed requests.

4. **Receiver:** Fetches the payload, uses the `secret` from their URL hash to decrypt, and matches the `userId` against the member directory.

## **4. Technical System Design**

### **4.1 Frontend (React PWA + Vite)**

- **Web Crypto API:** Handles AES-GCM for messages, PBKDF2 for the `roomAccessKey`, and **HMAC-SHA-256** for generating the request signatures.
- **Identity Management:** Stores the server-assigned `userId`, `roomAccessKey`, `join_eid`, and local `Display Name` in `localStorage`.
- **Message Sync:** On reconnect, the client requests events where `eid > lastSeenEid`.

### **4.2 Backend (Go + Gin + coder/websocket)**

- **Room Management:** Handles `roomId` generation and expiry logic.
- **Boost Processing:** Applied via payment webhook (SePay / Paddle). Runs an atomic Lua script that updates `expires_at`, `max_participants`, and `max_events` in meta, appends a `boosted` system event to the event log, and re-expires all room keys using `PEXPIREAT`. Broadcasts a `room:boosted` WebSocket event to all connected clients.
- **Message Ordering:** Uses an atomic Redis `INCR` on `room:{roomId}:event_seq` to assign a monotonically increasing `eid` to every incoming verified event.
- **Access Control:** Rejects join requests if the member registry is at capacity.
  - **Middleware:** Validates every incoming message/request by checking the `authToken` signature and timestamp window.
- **User Management:** Handles `userId` generation and records the starting message ID for that user.

## **5. Development Roadmap (MVP)**

### **Phase 1: The "Blind" Backbone**

- Server-side `roomId` generation.
- Client-side URL Hash logic and Web Crypto implementation (AES-GCM + PBKDF2).
- Redis integration for message history with **sequential ID generation**.

### **Phase 2: The "Identity" Layer**

- Server-side `userId` generation and Display Name registry.
- **Implementation of the HMAC signing middleware and 5s timestamp validation.**
- Logic to filter history retrieval based on `join_eid`.

### **Phase 3: PWA & Notifications**

- Manifest configuration and Web Push integration.

### **Phase 4: Binary Sharing (P2P)**

- WebRTC DataChannel implementation for images and location.

## **6. Business Logic & Constraints**

| **Feature**          | **Free**             | **Plus Boost**       | **Pro Boost**        |
| -------------------- | -------------------- | -------------------- | -------------------- |
| **Participant Cap**  | 5 **Total** Users    | 20 **Total** Users   | 100 **Total** Users  |
| **Room Lifetime**    | 1 Hour               | 24 Hours             | 7 Days               |
| **Message Buffer**   | 50 Events            | 100 Events           | 200 Events           |
| **Binary Sharing**   | P2P Only             | P2P Only             | P2P Only             |
| **Group Name**       | Static (at creation) | Static (at creation) | Static (at creation) |
| **Pricing**          | $0                   | One-time Fee / Room  | One-time Fee / Room  |

**Boost Stacking Rules:**
- **Expiry:** Each boost adds its full duration to the current `expires_at`. Boosting early is never penalized.
- **Participant / Event cap:** `MAX(current, boost value)` — purchasing a weaker boost after a stronger one does not reduce caps.
- **Boost options:** Served dynamically from the backend (`GET /api/boost-options`). Names and values can change without a client update.
- **Who can boost:** Any current member, or any user with the room `secret` (via QR/link) attempting to join a full room where a boost would raise the participant cap above the current count.
